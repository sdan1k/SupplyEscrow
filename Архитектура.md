# Архитектура системы SupplyEscrow (Обновленная версия)

## 1. Обзор архитектуры

Система `SupplyEscrow` построена на **гибридной архитектуре**, интегрирующей смарт-контракты Ethereum-совместимых сетей, минималистичный фронтенд и симулированные внешние системы. Архитектура спроектирована для автоматизации корпоративных расчётов через ЦФА в соответствии с 11-шаговым юридическим меморандумом и требованиями ФЗ-259.

### 1.1. Ключевые архитектурные принципы

| Принцип | Реализация | Обоснование |
|---------|------------|-------------|
| **Нулевое доверие (Zero Trust)** | Каждый шаг верифицируется кодом смарт-контракта, состояние неизменяемо | Исключает необходимость доверия между контрагентами, соответствует блокчейн-философии |
| **Юридическая корректность** | Разделение учёта (ЦФА) и расчётов (RUB), запрет прямого трансфера ЦФА поставщику | Соответствие ФЗ-259 "О ЦФА", предотвращение переквалификации в платёжное средство |
| **Разделение ответственности** | ОИС управляет ЦФА, смарт-контракт — статусами, банк — рублёвыми переводами | Распределение рисков, соответствие регуляторным требованиям |
| **Масштабируемость через изоляцию** | Каждый контракт — независимый инстанс, многоконтрактный UI | Поддержка множества параллельных сделок без взаимного влияния |
| **Прозрачность и аудируемость** | Все события фиксируются в блокчейне, доступны для регуляторов | Соответствие требованиям корпоративного compliance и AML/CFT |

### 1.2. Инновационность архитектурного решения

Архитектура `SupplyEscrow` представляет собой **нетривиальное сочетание**:
1. **Блокчейн-ориентированной бизнес-логики** с юридически выверенными переходами состояний
2. **Гибридной модели учёта** (ЦФА в блокчейне + рублёвые расчёты через традиционные банки)
3. **Многоконтрактной изоляции** с единым интерфейсом управления
4. **Симуляции реальных интеграций** (ККТ, ГЛОНАСС, маркировка) в рамках MVP

Это **первая в России архитектура**, реализующая полный цикл автоматизации корпоративных расчётов через ЦФА с соответствием 259-ФЗ.

## 3. Компоненты архитектуры (фактическая реализация)

### 3.1. Фронтенд (`frontend/`) — минималистичный, но функциональный

**Технологический выбор**: Чистый HTML/CSS/JavaScript без фреймворков
- **Обоснование**: Минимизация зависимостей, быстрый запуск, лёгкая отладка
- **Инновационность**: Многоконтрактный state management на ванильном JS

**Ключевые файлы**:
- `index.html` (460 строк) — структура интерфейса с семантической разметкой
- `app.js` (540 строк) — бизнес-логика, управление состоянием, интеграция с Web3
- Стили инлайн — минимизация файлов

**State Management (оригинальная реализация)**:
```javascript
// Многоконтрактное состояние
let contracts = []; // Массив активных контрактов
let activeContractId = null; // Текущий выбранный контракт

// Структура контракта (соответствует юридическому меморандуму)
{
  id: number,
  contractAddress: string,
  // ЦФА-ориентированные поля
  rubleAmount: number,      // RUB в ОИС
  cfaIssued: number,        // ЦФА выпущены
  cfaBlocked: number,       // ЦФА заблокированы
  cfaStatus: 'BLOCKED'|'READY_FOR_REDEMPTION'|'REDEEMED',
  // Этапы 11-шагового процесса
  paymentOrderCreated: boolean,   // Шаг 7
  paymentOrderExecuted: boolean,  // Шаг 8
  cfaRedeemed: boolean,          // Шаг 9
  // Проверки и compliance
  markingChecked: boolean,        // Честный ЗНАК
  depositConfirmed: boolean,      // Подтверждение ОИС
  // Арбитраж
  redemptionSuspended: boolean    // Приостановление при диспуте
}
```

**Инновационные аспекты фронтенда**:
1. **Многоконтрактный интерфейс**: Боковая панель с динамическим переключением
2. **Ролевая модель**: 3 предопределённые роли (OIS Operator, Buyer, Supplier) с разными правами
3. **Демо-автомат**: 11-шаговый сценарий с задержками, имитирующий реальный процесс
4. **Валидация данных**: Правило двух источников (ККТ + ГЛОНАСС) с порогом 5%

### 3.3. Интеграционные интерфейсы (симуляция)

**Принцип симуляции**: Реальные интеграции заменены кодом, имитирующим поведение внешних систем.

| Система | Симуляция в коде | Реальный протокол (для production) |
|---------|------------------|-------------------------------------|
| **ОИС API** | `contract.cfaIssued`, `contract.cfaRedeemed` | REST/WebSocket с JWT-авторизацией |
| **ККТ оракул** | `simulateAPIData()` с генерацией случайных данных | HTTPS API с подписью запросов |
| **ГЛОНАСС оракул** | Расчет volume по координатам, валидация | MQTT/WebSocket поток данных |
| **Банковский шлюз** | Логи "Поручение исполнено банком" | ISO 20022, API банков (Тинькофф, Сбер) |
| **Честный ЗНАК** | Проверка `productMarkingId` | HTTPS API ГИС МТ (закрытый) |

**Обоснование подхода симуляции**:
1. **MVP-фокус**: Демонстрация работоспособности логики без сложных интеграций
2. **Изоляция рисков**: Отсутствие зависимостей от внешних API на этапе демо
3. **Гибкость тестирования**: Возможность эмуляции различных сценариев (успех/сбой)

## 5. Технологический стек и обоснование выбора

### 5.1. Блокчейн и смарт-контракты

| Технология | Версия | Обоснование выбора | Альтернативы и причины отказа |
|------------|--------|-------------------|--------------------------------|
| **Solidity** | 0.8.20 | Наиболее зрелый язык для Ethereum, полная поддержка tooling, backward compatibility | Vyper (меньше сообщество), Rust (нестандартно для EVM) |
| **Hardhat** | 2.19.0 | Лучшая среда разработки для Ethereum: тестирование, отладка, развёртывание | Truffle (устаревший), Foundry (меньше плагинов) |
| **OpenZeppelin** | 5.2.0 | Стандартные безопасные реализации, аудированные сообществом | Самописные реализации (высокий риск уязвимостей) |
| **Ethers.js** | 6.13.5 | Современная, типобезопасная библиотека для Web3 | Web3.js (устаревшая архитектура) |

**Инновационность стека**:
- Использование **Solidity 0.8.20** с встроенной защитой от переполнения (без SafeMath)
- **TypeChain** для генерации TypeScript-типов из ABI — повышение безопасности разработки
- **Hardhat Upgrades Plugin** для поддержки обновляемых прокси-контрактов (задел на будущее)

### 5.2. Фронтенд и инфраструктура

| Технология | Обоснование | Производственная альтернатива |
|------------|-------------|-------------------------------|
| **Vanilla JS** | Минимизация зависимостей, быстрый запуск, полный контроль | React/Vue + состояние (Redux/Vuex) для сложных приложений |
| **Python HTTP Server** | Простейший способ раздачи статики, встроен в ОС | Nginx/Apache для production, Node.js Express для SSR |
| **Hardhat Node** | Локальная разработка с предзаполненными аккаунтами | Ganache (устарел), Anvil (Foundry) |

**Архитектурное решение по фронтенду**:
- **Отказ от фреймворков**: Уменьшение bundle size, отсутствие виртуального DOM
- **Многоконтрактное состояние на чистом JS**: Демонстрация возможности без избыточных абстракций
- **Интеграция ethers.js напрямую**: Минимизация слоёв абстракции

### 5.3. Тестирование и качество

| Инструмент | Назначение | Обоснование |

| **Hardhat/Waffle** | Unit-тесты смарт-контрактов | Интеграция с Hardhat, поддержка TypeScript |
| **Chai** | Assertion library | Человекочитаемые assertions, большая community |
| **Hardhat Gas Reporter** | Анализ стоимости газа | Оптимизация для production-развёртывания |
| **Solidity Coverage** | Покрытие кода тестами | Метрика качества для аудита |

## 6. Аспекты безопасности архитектуры

### 6.1. Безопасность смарт-контрактов

**Реализованные меры**:

| Угроза | Мера защиты | Реализация |
|--------|-------------|------------|
| **Reentrancy** | Checks-Effects-Interactions | Во всех external функциях |
| **Integer overflow** | Solidity 0.8 встроенная защита | Отказ от SafeMath |
| **Access control** | Модификаторы onlyRole, OpenZeppelin AccessControl | `onlyBuyer`, `onlyOperator`, etc. |
| **Input validation** | Require statements для параметров | Проверка сумм, статусов, прав |
| **Front-running** | Статусные переходы, нет аукционов | Неприменимо для бизнес-логики |

**Аудит и верификация**:
- Использование **аудированных библиотек OpenZeppelin**
- **Модификаторы доступа** для всех критических функций
- **Event logging** для отслеживания изменений состояния
- **Тестовое покрытие** основных attack vectors

### 6.2. Безопасность фронтенда и данных

| Аспект | Реализация в MVP | Production-рекомендации |
|--------|------------------|-------------------------|
| **Хранение ключей** | В памяти браузера (ethers.js) | Hardware wallet (Ledger/Trezor), MPC-решения |
| **Подключение к сети** | Локальный Hardhat node | WSS-провайдеры (Alchemy, Infura) с API keys |
| **CSRF/XSS защита** | Минимальна (статический контент) | CSP headers, JWT tokens, CORS policy |
| **Данные пользователей** | Только публичные адреса | Шифрование PII, GDPR compliance |

### 6.3. Конфиденциальность и соответствие

**Соответствие требованиям**:
1. **ФЗ-259 "О ЦФА"**: Разделение учёта и расчётов, запрет трансфера ЦФА
2. **ФЗ-115 "О ПОД/ФТ"**: Роли AML Officer, логирование, KYC (симуляция)
3. **152-ФЗ "О персональных данных"**: Минимизация PII, только публичные адреса

**Архитектурные решения для конфиденциальности**:
- **Псевдонимизация**: Использование blockchain addresses вместо персональных данных
- **Минимизация данных**: Хранение в блокчейне только необходимой информации
- **Слоистая архитектура**: Separation of concerns для compliance-критических компонентов

## 7. Аспекты надёжности и отказоустойчивости

### 7.1. Надёжность блокчейн-слоя

| Аспект | Реализация | Обоснование |
|--------|------------|-------------|
| **Неизменяемость состояния** | Все изменения через транзакции, события immutable | Гарантия аудируемости, невозможность подделки истории |
| **Восстановление после сбоев** | Состояние сохраняется в блокчейне, можно replay | Отказоустойчивость на уровне данных |
| **Консистентность** | Детерминированные переходы статусов | Предсказуемость поведения системы |

### 7.2. Отказоустойчивость интеграций

**Паттерн "Правило двух источников"**:
```javascript
// Валидация данных поставки
const volumeDiff = Math.abs(kktData.volume - gnssData.calculatedVolume);
const maxDiff = kktData.volume * 0.05; // 5% порог

if (volumeDiff > maxDiff || !gnssData.measurementValid) {
    // Автоматический переход в диспут
    contract.arbitrationStarted = true;
    contract.redemptionSuspended = true;
    addLog('❌ Расхождение >5% - требуется ручная проверка', 'error');
}
```

**Fallback-механизмы**:
1. **Оракул недоступен**: Таймаут → ручная проверка оператором
2. **Расхождение данных**: Автоматический диспут → арбитраж
3. **Сбой ОИС**: Приостановление операций → manual override

### 7.3. Резервные сценарии (Dispute Flow)

Архитектура предусматривает **деградацию функционала** при сбоях:
- Автоматический режим → полуавтоматический (диспут) → полностью ручной (арбитраж)
- Каждый уровень добавляет человеческое вмешательство, но сохраняет аудируемость

## 8. Аспекты производительности и масштабируемости

### 8.1. Производительность смарт-контрактов

**Оптимизации газа**:
```solidity
// hardhat.config.js
settings: {
    optimizer: {
        enabled: true,
        runs: 200  // Баланс между размером кода и стоимостью выполнения
    }
}
```

**Метрики производительности (локальная сеть)**:
- Создание контракта: ~3-5 секунд
- Подтверждение депозита: ~2-3 секунды
- Подтверждение доставки: ~2-4 секунды
- Открытие спора: ~1-2 секунды

**Ограничения и планируемые оптимизации**:
- Текущая реализация: O(1) для большинства операций
- Планируется: Батчинг операций для массовых выплат
- Планируется: Использование EIP-712 для off-chain подписей

### 8.2. Масштабируемость архитектуры

**Горизонтальное масштабирование**:
1. **Многоконтрактная изоляция**: Каждая сделка — независимый контракт
2. **Шардирование по типам контрактов**: SupplyEscrow vs CorporateEscrow
3. **Кэширование read-only данных**: Статусы, балансы, история

**Вертикальное масштабирование**:
1. **Оптимизация газа**: Уменьшение сложности вычислений
2. **Компрессия данных**: Использование bytes32 вместо string где возможно
3. **Off-chain вычисления**: EIP-712 для сложных валидаций

### 8.3. Производительность фронтенда

**Оптимизации**:
- **Минимизация перерисовок**: Точечные updates UI при изменениях состояния
- **Виртуализация списков**: Для >100 контрактов (не реализовано в MVP)
- **Lazy loading**: Динамическая загрузка данных контракта при выборе

## 11. Оценка архитектурных решений

3. **Многоконтрактная изоляция**: Каждая сделка — независимый security domain
4. **Симуляция реальных интеграций**: Демонстрация полного цикла без внешних зависимостей
